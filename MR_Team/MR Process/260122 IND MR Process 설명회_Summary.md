# [인디아 본부 마일스톤 리뷰 프로세스 설명회] 상세 강의 노트

## 1. 개요
- **일시:** 2026.01.22 (목) 오전 10:32 (약 57분)
- **참석자:** MR팀(지혜), 인디아 본부(조휘연, 낙형 님 등)
- **목적:** 인디아 본부의 크리켓 IP 라이센싱 의사결정을 위한 마일스톤 리뷰 프로세스 설명

## 2. 상세 분석/학습 내용 (주제별 정리)

### Section 1. 마일스톤 리뷰 프로세스의 본질과 철학

#### 1.1 경영 프로세스로서의 마일스톤 리뷰
- **핵심 개념:** 마일스톤 리뷰는 **'개발 숙제 검사'가 아니라 '경영 프로세스'**이다
- 일반 게임 회사의 마일스톤 리뷰: 개발 숙제를 했는지 안 했는지, 잘했는지 못했는지를 검사하는 과정
- **크래프톤의 마일스톤 리뷰:** 매번의 리뷰가 **비즈니스 피치(Business Pitch)**라고 생각해야 함
- **이유:** 크래프톤은 서브컬처 게임부터 모바일 퍼즐 게임까지 너무 많은 장르를 너무 많은 플랫폼에서 만들기 때문에 **"이렇게 만들어야 해"라는 통일된 기준이 없음**
- NC나 넷마블처럼 비슷한 게임을 찍어내는 회사와는 근본적으로 다름
- **결과적으로:** 개발팀이 "저희는 이렇게 검증하는 게 맞다고 생각합니다"라고 주장해야 하고, 대표님도 이 과정을 통해 고민하시는 구조

#### 1.2 Go/No-Go 의사결정의 목적
- **목적:** 대표님이 Go/No-Go 의사결정을 통해 **전사 자원 분배의 타당성을 검토하고 제고**하는 것
- 마일스톤 리뷰 통과 = 개발팀에게 개발 버짓을 준다는 의미
- **따라서:** "이 돈을 여기에 투자할 만한가?"를 판단하는 미팅

#### 1.3 의사결정 철학: 합의가 아닌 선택
- **의장님 말씀:** "의사결정은 합의가 아니다. 의사결정은 책임을 가진 대표자가 하면 되는 것"
- 의사결정이 잘 되기 위한 조건: **다양한 의견이 명확하게 드러나는 환경**
- 옳고 그름을 가리는 것도 아니고, 합의하는 것도 아니고, 설득하는 것도 아님
- **언론 비유:** 어떤 사건이 있을 때 왼쪽 사람들, 오른쪽 사람들의 의견을 모두 들어봐야 프레임을 넓혀서 좋은 의사결정을 할 수 있음
- MR팀의 역할: **의사결정을 위한 환경을 만들어 드리는 것**

### Section 2. 프로세스 거버넌스 및 운영 체계

#### 2.1 기획과 운영의 분리
- **기획:** 기조실에서 담당 (전체 회사 방향성, 여러 부서 인볼브먼트 고려)
- **운영:** MR팀(마일스톤 리뷰 사무국)에서 담당
- **연혁:** 초기에는 의장님이 오너였으나 현재는 기조실로 이관됨
- **현재 버전:** 2.0 (2.01 버전 대표님 승인 대기 중)
- 주기적으로 피드백 받아 업데이트하는 프로젝트

#### 2.2 프로세스 설계 철학
- **의도적으로 비워둔 영역 존재:** 너무 많은 것들을 따박따박 프로세스화하면 "프로세스 뒤에 숨는 사람들"이 생김 (대표님 언급)
- **동일한 의견이 나오는 것이 오히려 문제:** 서로 눈치 보지 않고 따로 떨어뜨려서 다양한 의견이 나오도록 설계

#### 2.3 스콥(Scope) 정의: 어떤 프로젝트가 대상인가?
- **기본 원칙:** 마일스톤 리뷰의 의사결정자는 대표님
- **예외 케이스들:**
  - Flyway Games: 독립적 운영 목적으로 설립 → 마일스톤 리뷰 대상 아님
  - 펍지 스튜디오: 펍지 IP 프랜차이즈에서 퍼블리싱하는 프로젝트 → 기본적으로 대상 아님 (단, 대표님이 장태석 님에게 "한번 써봐라" 해서 일부 진행)
  - 인디아 오피스: 자체적으로 사업 진행/결정하는 오피스 → 기본적으로 대상 아님
- **크리켓 건 특수성:** 현일 님이 의사결정자이므로, **의사결정 프레임워크로서** 이 건에 한해 진행 (향후 인디아 모든 게임에 적용할지는 별건)

### Section 3. RAPID 의사결정 프레임워크와 역할 구조

#### 3.1 RAPID 프레임워크 개요
| 역할 | 설명 | 구속력 |
|------|------|--------|
| **I (Input)** | 인풋/의견 제공자 | **없음** - 의견만 제공 |
| **A (Agree)** | 동의 필요 역할 (예: 법무팀) | **있음** - 구속력 보유 |
| **R (Recommend)** | 제안자 - 주장하고 책임지는 역할 | 의사결정 수용 여부 결정권 |
| **D (Decide)** | 의사결정자 | 최종 결정권 |
| **P (Perform)** | 실행자 | 제안 실행 |

#### 3.2 I(Input)의 비구속성이 중요한 이유
- **핵심:** Input 제공자의 의견에 구속력이 없어야 함
- 만약 R이 I의 의견을 받아야 한다면 → 그건 Align(합의)이 됨
- 그러면 책임이 모호해짐: "그때 인풋 제공자가 말해서 받은 건데요"라고 회피 가능
- **따라서:** I의 내용을 받을지 안 받을지는 R이 결정 → R에게 권한과 책임 부여

#### 3.3 크리켓 프로젝트 역할 배분
- **R (제안자):** 낙형 님 (제작 PD) - "이 게임은 IP 라이센싱 2천억 계약을 할 만큼의 준비가 됐다"를 주장해야 하는 역할
- **D (의사결정자):** 현일 님 (인디아 본부장)
- **챔피언:** 현일 님 (의사결정자와 동일 - 특수 케이스)
- **S2:** 없음 (인디아 프로젝트 특수성)
- **담당 퍼블리싱 A역할:** 인디아 퍼블리싱 (이민우 님 담당)

#### 3.4 게임성 검증 vs 사업성 검증
| 구분 | 게임성 검증 | 사업성 검증 |
|------|-------------|-------------|
| **시점** | 풀 프로덕션 전 | 풀 프로덕션 후 |
| **제안자 (R)** | 개발팀 PD | 사업 책임자 |
| **주요 비용** | 개발 비용 | 마케팅/인력 투입 비용 |
| **피칭 내용** | 게임 개발 현황/계획 | 퍼블리싱 전략/예산/인력 |

- **크리켓 건:** 게임성 리뷰로 진행 (인디아 퍼블리싱이 자체 검증 가능하므로)
- **중요 포인트:** 게임성 검증 단계에서 개발팀이 "400만 장 팔 수 있다"고 주장해도, 담당 퍼블리싱은 "저는 200만 장밖에 못 팔 것 같습니다"라고 다르게 의견 낼 수 있음 → 본인 주장에 대한 책임 구조

#### 4.1 인풋 항목 분류
| 분류 | 항목 | 설명 |
|------|------|------|
| **Key Success Factor** | 핵심 성공 요소 | 개발팀이 주장하는 검증 목표 |

  - 외주 계약의 지속성
  - 딜의 지속성
- **제작팀 실행력:** 낙형 님이 작성 - 외주 회사의 계획 대비 실제 생산력, 이벤트 제때 만들 수 있는지 등

#### 4.3 빌드 리뷰 담당 조직 (3개 팀)
| 조직 | 소속 | 역할 |
|------|------|------|
| **게임 애널리시스 팀** | 게임 인사이트 디파트먼트 | 빌드 리뷰 |

#### 4.4 기대치 제공자 vs 인풋 제공자
| 구분 | 기대치 제공자 | 인풋 제공자 |
|------|---------------|-------------|

| **예시** | "매출 천억 달성할 거라는데 저희는 천억 안 될 것 같은데요" | "이렇게 만들면 이런 재미가 나올 거라는데 저희 생각은 좀 다릅니다" |

### Section 5. 프로세스 타임라인 및 절차

#### 5.1 전체 타임라인 (약 6~8주)
| 시점 | 활동 |
|------|------|
| **8주 전** | 프로세스 설명, 일정/작성 내용 공유 |
| **6주 전** | 빌드 수령, 유저 테스트 시작 |
| **5주 전** | 인풋 조직 배포, 개발팀 QnA 세션 |
| **4주 전** | 인풋 마감 → 개발팀에 전체 공유 |
| **3주 전** | 기대치 작성 시작 |
| **2주 전** | 유저 테스트 마감, 기대치 마감 → 개발팀 공유 |
| **1주 전** | 개발팀 최종 문서 마감, 대표님께 전체 자료 공유 |
| **D-Day** | 마일스톤 리뷰 미팅 |
| **+10일 이내** | 서면 의사결정 공유 (R, Finance, 개발팀 등) |

#### 5.2 핵심 절차 상세
1. **인풋 항목 구성:** S2(없는 경우 인디아 본부)와 논의하여 필요 인풋 결정 → 대표님(현일 님) 확인
2. **빌드 배포 및 QnA:** 인풋 조직에 빌드 배포 후 1주 뒤 개발팀 QnA 세션 (오해 방지, 키 퀘스천 해소)
3. **인풋 → 기대치 순서 이유:** 기대치 작성자가 모든 장르를 아는 것이 아니므로, 인풋(장르 분석, 빌드 분석, 유저 테스트 등)을 먼저 보고 기대치 작성
4. **개발팀 최종안:** 인풋+기대치 모두 받은 후, 답변이 필요한 질문을 픽해서 답변 포함한 최종 발표 자료 준비

#### 5.3 인력 기준에 따른 마일스톤 리뷰 시점
| 인력 규모 | 필요 마일스톤 리뷰 |
|-----------|-------------------|
| **20명 이상** | 킥오프 마일스톤 리뷰 |
| **50명 이상** | 풀 프로덕션 승인 (게임성 검증 완료) |

- **원칙:** 비용이 갑자기 훅 올라가기 전에 승인을 받아라

### Section 6. 유저 테스트 관련 논의

#### 6.1 UGA 팀을 통한 테스트 권장
- **권장 이유:**
  - 테스트 방법에 따라 결과가 매우 다를 수 있음
  - 한 조직이 일관된 방법으로 하면 비교 지표로 활용 가능
  - UGA 팀은 **심리학 석사 출신** 위주로 구성 - 학문적 백그라운드, 통계 처리, 방법론 전문성 보유
- **외주 테스트의 문제점:** 외주에서 리크루팅한 사람들도 웬만하면 좋게 평가하는 경향
- **인디아 건 특수성:** 의사결정자가 현일 님이므로 현일 님이 믿을 만한 구성으로 하는 게 깔끔

#### 6.2 인디아 시장 특수성 대응
- **문제점:** 빌드 리뷰 담당자들이 인디아 시장/크리켓 장르를 잘 모름
- **대응 방안:**
  - 사전에 자료 공유 (리서치 페이퍼, 정리된 자료 등)
  - QnA 세션 또는 자료 공유 형태로 인풋 조직에 사전 정보 제공
  - 스포츠 게임 특성상 "축구를 안 보던 사람이 축구 게임 재미를 판단하기 어려움" - 사전 리서치 시간 필요

### Section 7. 크리켓 프로젝트 특수 논의 사항

#### 7.1 아젠다 명확화 필요성
- **현재 상황:** 인디아 본부에서 4개의 크리켓 게임 개발 중
  1. 리얼 크리켓 (노틸러스)
  2. 크리켓 올스타즈 (Round Six 외주)
  3. 신규 외주 개발팀 프로젝트
  4. 리얼 크리켓 맥스 (노틸러스)
- **IP 계약:** 크래프톤 전체 크리켓 게임에 사용 가능하도록 계약 예정 → 개별 게임 성패와 무관
- **핵심 질문:** "이 마일스톤 리뷰의 아젠다가 뭔가?"
  - 옵션 A: 4개 중 어떤 게임에 IP를 붙일 것인가? (종합적 평가 필요)
  - 옵션 B: 현재 진행 중인 빌드의 퀄리티가 준수한가?

#### 7.2 인디아 본부의 기존 인식 vs 실제
| 기존 인식 | 실제 |
|-----------|------|
| 크래프톤에서 진행되는 프로젝트는 무조건 마일스톤 리뷰 필수 | 의사결정 프레임워크로서 선택적 활용 |
| 3개월마다 개발 마일스톤마다 빌드 제출 | 특정 의사결정이 필요할 때 진행 |
| 정밀한 스코어링을 위한 프로세스 | Go/No-Go 의사결정에 최적화된 프로세스 |

#### 7.3 후속 조치
- 낙형 님, 현일 님과 **아젠다 명확화** 논의 필요
- 아젠다에 따라 인풋 종류, 시기, 어떤 빌드로 리뷰할지 결정
- 메일로 아젠다 공유 → 시기 결정 → 월간 미팅 셋업 논의

## 3. Q&A 및 부연 설명

**Q: 유저 테스트를 인디아 본부에서 외주로 진행해도 되나요?**
- A: 결과를 인풋으로 쓸 수는 있으나, HQ에서 진행하는 마일스톤 리뷰에서는 UGA 팀 테스트가 기본 필수. 다만 이 건은 인디아 본부 진행이므로 현일 님이 신뢰할 수 있는 구성으로 결정하면 됨.

**Q: 마일스톤 리뷰 일정 조정이 가능한가요?**
- A: 가능하나 **대표님 승인 필요**. 마음대로 일정 변경 불가.

**Q: 의사결정 결과는 어떤 종류가 있나요?**
- A: 승인, 조건부 승인(킥오프만), 재도전(약속 미충족이나 드랍은 아님), 드랍

## 4. 요약 및 결론

**마일스톤 리뷰 프로세스의 핵심은 '개발 검증'이 아닌 '경영 의사결정 지원'에 있다.** 이 프로세스는 다양한 전문가들의 의견을 체계적으로 수집하여 의사결정자가 충분한 정보를 바탕으로 Go/No-Go를 판단할 수 있는 환경을 만드는 것이 목적이다. 인풋 제공자들의 의견에는 구속력이 없으며, 최종 결정과 책임은 의사결정자에게 있다.

**크리켓 프로젝트의 경우**, 인디아 본부의 특수성(자체 퍼블리싱 조직 보유, 외주 개발, 의사결정자가 현일 님)을 고려하여 게임성 리뷰로 진행하되, **먼저 "이 마일스톤 리뷰를 통해 무엇을 의사결정할 것인가"라는 아젠다를 명확히 정의**하는 것이 선행되어야 한다. 4개의 크리켓 게임 중 어디에 IP를 붙일지를 결정하는 것인지, 특정 빌드의 퀄리티를 검증하는 것인지에 따라 필요한 인풋의 종류와 참여 조직이 달라지기 때문이다.

**후속 액션으로는** 낙형 님, 현일 님과의 아젠다 협의 후 MR팀에 메일로 공유하면, MR팀에서 적절한 시기와 월간 미팅 셋업을 논의하게 된다. 인디아 시장 및 크리켓 장르의 특수성을 고려하여 인풋 조직들에게 사전 자료 공유와 QnA 세션을 통해 충분한 배경 지식을 제공하는 것이 중요하다.

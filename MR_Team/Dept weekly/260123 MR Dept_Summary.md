### Section 5. Steam 리뷰 분석 대시보드 개발

#### 프로젝트 개요

- 수집한 정보를 기반으로 추출 가능한 사이트를 대시보드로 구현

#### 궁극적 목표
- 특정 게임뿐 아니라 **여러 게임 비교 분석**
- 게임별 특이점 파악 가능하도록 확장

#### Steam 리뷰 메타데이터 활용
- 뽑아낼 수 있는 정보가 매우 많음
- 대부분의 정보를 **자동화 + 시각화**하면:
  - 진엽 님의 게임 경험 내용 + 데이터 분석을 **한 통으로 묶어** 볼 수 있음
  - 레퍼런스 게임 분석 시 **토픽 모델링 결과**까지 첨부 가능
  - **경험 + 데이터 분석이 통합된 문서** 생성 목표

#### 활용 방안
- 구승 님 말씀: "우리 팀이 어떤 형태의 결과물을 만들어낼 수 있는가"를 설명할 때 사용
- 팀 정체성 설명용 / 경험과 분석의 가치 있는 결과물 증명

### Section 6. PBB 포스트모템 및 HTML 보고서 도구 개발

#### PBB 포스트모템 주요 문제
- **대량 데이터를 HTML 하나의 보고서에 담아야 함**
- AI가 한 번에 오류 없이 작업하기 어려움
- **교훈**: 
  - 시작부터 **자료 관리** 및 **HTML 스켈레톤 분리** 작업 필요
  - 전반 **파이프라인 설계**적 접근 필수

#### 개발한 도구 1: HTML 분할/통합 도구
- **문제**: 규모가 커지면 AI가 뻗거나 오류 발생 → 파일 분할 작업 필요
- **해결**: 
  - 하나의 큰 파일을 **자동으로 분할**
  - 쪼개진 파일 각각 수정 → **통합 트리뷰 자동 생성**
  - 업데이트 없이도 통합 본을 바로바로 보면서 작업 가능

#### 개발한 도구 2: JSON→HTML 매핑 도구 (보수 공사 중)
- **목적**: 정해진 HTML 폼(샘플)에 분석 데이터를 자동 매핑
- **작동 방식**:
  - JSON 파일의 패턴 인식 → HTML 필드와 연결
  - 탭, 아코디언 등 복잡한 구조도 기계적으로 빠르게 처리
- **활용 케이스**: 
  - 유저 반응/커뮤니티 반응 조사 보고서 폼이 정해지면
  - 내용만 바뀌는 보고서를 원하는 폼대로 자동 생성

#### 가이드라인 문서화
- HTML/JSON 제작 시 **규칙과 기조**를 포함한 문서 정리 예정
- 툴 가이드라인 포함 → 차주 상혁 님께 공유 후 피드백 → 실 공유

#### 세령 님 보충 설명
- **도구 1 등장 배경**: 커서 등에서 대량 텍스트 작업 시 뻗는 문제 → 분할 후 합치는 작업 공수 절감
- **도구 2 등장 배경**: 
  - 보고서 샘플 폼 유지 + 데이터만 교체
  - JSON에서 HTML로 이관 시 샘플 형식 그대로 데이터 확장

### Section 8. Claude Skills 심층 학습

#### Claude Skills 개념
- **정의**: 미리 정의된 인스트럭션(규칙/지침)을 파일로 공유하는 형태
- **핵심 차별점**: 
  - GPTs/Gems는 프로젝트마다 분리 → 다른 채팅에서 사용 불가
  - **Skills는 한 채팅창에서 여러 스킬 조합 사용 가능**

#### Skills vs Agent 구분 (중요)
| 구분 | Skills | Agent |
|------|--------|-------|
| 본질 | **인스트럭션 (업무 지시서)** | **전문가 역할 (사람)** |
| 내용 | MD 파일로 규칙/지침 정의 | 마케팅 전문가, 개발 전문가 등 역할 부여 |
| 코드 포함 | 참고용 코딩 규약으로 포함 가능 | 에이전트가 스킬을 참조해서 실행 |
| 연결 방식 | 스킬 간 직접 연결 개념 없음 | 에이전트가 스킬을 참조, 순차/병렬 실행 가능 |
| 멀티 개념 | 없음 (단일 인스트럭션) | 멀티 에이전트 가능 (오케스트레이터 + 하위 에이전트) |

#### Skills 내 코드의 의미
- 스킬 안의 코드는 **실행되는 코드가 아님**
- **"너 코딩할 때 이 코드처럼 짜"** = 코딩 규약/참고 코드
- 예: 클래스 구조, 파일명 규칙, 주석 언어 등

#### Skills 설치 및 사용
- **설치 방법**: CMD에서 설치 명령 → 어디에 설치할지 선택 (커서, 글로벌 등)
- **커서에서**: 커서 룰즈(Cursor Rules)에 스킬 MD 파일 추가
- **조합 사용**: 100개 스킬 중 필요한 5개만 골라서 사용 가능

#### Skills 공유 생태계
- **GitHub**: "Claude Skills"로 검색하면 다양한 스킬 공개
- **smithery.ai**: 인기/트렌딩 스킬 큐레이션 사이트
- **awesome-claude-skills**: 다양한 스킬 모음 리포지토리
- **목적**: 포트폴리오 / 오픈소스 기여 / 팀 협업

#### Skill Creator 활용
- 스킬 만들 때 **질문을 통해 가이드** 제공
- "어떤 스킬을 만들고 싶어"라고 하면 → 필요한 질문들을 줌
- **"우리가 뭘 모르는지 모르니까 못 쓰는 것"** → 스킬 크리에이터가 해결

#### Claude의 전략적 차별점
- **MCP(Model Context Protocol)**: Claude가 만들어 표준화
- **Skills**: Claude가 만들어 생태계 확장
- **특징**: 만들고 → 많이 쓰이면 → 그냥 표준으로 오픈 → 생태계 장악
- **GPT와 차이**: GPT는 "우리가 짱" / Claude는 "생태계를 어떻게 먹을까" 고민

#### 구체적 방향
1. **Skills 공부** → 부서에 필요한 스킬 제작
2. **스킬 조합** → 에이전트로 묶기
3. **외부 배포**: 마일스톤 리뷰 가이드 등 → 스킬 기반 웹 서비스화 가능

#### 멀티 에이전트 예시 (용환 님 케이스)
- **오케스트레이터 에이전트**: 어떤 분석이 필요한지 자체 판단
- **하위 에이전트 6개**: 분석 종류별 전문화
- 오케스트레이터가 "이 주제는 3개 에이전트 필요" 판단 → 자동 분배 → 결과 수집

#### 개발팀 사례
- 팀장이 제일 먼저 하는 일: **Skills 만들기**
- **이유**: 10명이 AI로 작업하면 다 다른 코드가 나옴
- **해결**: 코딩 컨벤션(클래스 구조, 파일명, 주석 등)을 스킬에 정의 → 전파
- 결과: AI를 마음대로 쓰더라도 **규칙은 절대 안 깨짐** = 인수인계 효과

## 3. Q&A 및 부연 설명

### Q: Skills와 GPTs/Gems의 차이점은?
**A**: GPTs/Gems는 프로젝트마다 분리되어 다른 채팅에서 사용 불가. Skills는 한 채팅창에서 여러 스킬을 불러와 조합 사용 가능. 또한 Skills는 MD 파일로 공유되어 누구나 설치 가능.

### Q: Skills 안에 코드가 있으면 실행되는 건가?
**A**: 아니요. 스킬 안의 코드는 **코딩 규약/참고 코드**입니다. "이 코드처럼 짜라"는 인스트럭션이지, 실제 실행되는 코드가 아닙니다.

### Q: Skills와 Agent의 차이는?
**A**: Skills는 업무 지시서(인스트럭션), Agent는 전문가 역할. Agent가 Skills를 참조해서 작업을 수행합니다. Skills는 멀티 개념이 없고, Agent는 멀티 에이전트로 구성 가능합니다.

### Q: Claude만 Skills가 있는 이유는?
**A**: Claude가 사용성(UX)을 많이 고려하는 편. 개발자에게 어색하지 않은 부분도 일반인을 위해 개선. MCP, Skills 모두 Claude가 만들어 표준화한 것으로, 생태계 확장 전략의 일환입니다.

이번 MR Dept 미팅은 주간 업무 공유와 함께 **Claude Skills에 대한 심층적인 학습 세션**이 진행된 의미 있는 자리였습니다.

**첫째**, 각 프로젝트별로 KIZNA 마일스톤 리뷰 완료, PBB 최종 발표 자료 수급, 언노운월즈 S2 교체 등 주요 현황이 공유되었으며, 프로덕트 대시보드 취합이 50% 진행되어 마감일 팔로업이 필요한 상황입니다.

**둘째**, Steam 리뷰 분석 대시보드, HTML 보고서 자동화 도구 등 **AI를 활용한 업무 효율화 사례**가 구체적으로 공유되었습니다. 특히 PBB 포스트모템을 통해 대량 데이터 처리 시 파이프라인 설계의 중요성이 강조되었습니다.

**셋째**, Claude Skills의 개념, 설치 방법, Agent와의 차이점, 멀티 에이전트 활용 등 **심층적인 AI 학습**이 이루어졌습니다. Skills는 인스트럭션(지시서)이고 Agent는 전문가 역할이라는 명확한 구분이 중요합니다.

**넷째**, 향후 AI 활용 방향으로 1분기는 개인 생산성 향상, 2~3분기부터는 **조직 단위 공용화**로 전환할 계획입니다. 각자 전문화했던 영역을 Skills로 만들어 부서 내 협업 도구로 발전시키는 것이 목표입니다.

**다섯째**, 연말 AI 평가 항목에 대비하여 **AI 활용 성과 아카이빙**을 시작해야 하며, 결과 위주(시간 단축, 인사이트 획득 등)로 간결하게 기록하는 것이 권장됩니다.

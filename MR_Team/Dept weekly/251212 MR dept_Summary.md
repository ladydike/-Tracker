# [세션 제목: MR 부서 정기 미팅 및 프로젝트 현황 상세 강의 노트]

## 1. 개요
* **분석 대상:** 카나리아/서브노티카 모바일 MR 결과, 프로덕트 대시보드 보안 이슈, AI 크롤링 기술적 한계
* **세션 목적:** 주요 프로젝트의 의사결정 결과를 공유하고, 데이터 보안 및 자동화 툴 개발 과정에서의 기술적 장애물과 해결 방안을 논의함

## 2. 상세 분석 내용 (주제별 정리)

### Section 1. [마일스톤 리뷰(MR) 결과 및 기록 관리]
* **주요 의사결정:**
  * **카나리아(Canary):** 승인(Approve)으로 결론. 현재 최종 의사결정 초안 작성 중.
  * **서브노티카 모바일:** 최종 반려(Reject) 승인.
* **지식 관리(Wiki):**
  * **크리에이티브 위키:** 오리온(Orion) 등 신규 크리에이티브 프로젝트를 위한 위키 페이지 신설.
  * **통합 관리 원칙:** 유저 사용성 차원에서 **승인된 프로젝트와 반려된 프로젝트를 한곳에서 관리**하기로 함. 이는 과거 히스토리 파악을 용이하게 하기 위함임.

### Section 2. [프로덕트 대시보드 및 보안 정책]
* **DB 연동 이슈:** 웹앱을 애저(Azure)에 배포하는 과정에서 데이터 브릭스(Data Bricks) DB의 IP 접근 차단 문제 발생.
* **민감 정보 제한:** 보안 정책상 공용 DB에 PoC(챔피언, 에스투 등)의 **이메일, 전화번호 등 개인 정보 및 펍지(PUBG) IP 관련 정보 포함이 금지**됨.
* **해결 방안:** 
  * MVP(최소 기능 제품) 수준에서 불필요한 개인 정보는 제외함.
  * 보안 가이드에 따라 원천 데이터 읽기 권한이 있는 사람만 정보를 볼 수 있도록 권한 분기 처리.
  * 필요 시 회사 공용 DB 대신 별도의 자체 SQL DB 구축 검토.

### Section 3. [AI 자동화 및 기술적 제약]
* **크롤링 차단 이슈:** 전 세계 웹 트래픽의 약 80%가 AI 봇인 상황에서, 주요 사이트들이 봇 접근을 강력히 차단하고 있음.
* **기술적 한계:** RSS 피드를 통한 수집은 실시간성이 떨어지며(하루 10개 제한 등), 1시간 단위 폴링(Polling)도 하루 3~4회 이상 수행 시 차단되는 경우가 빈번함.
* **해결 방안:** 수집 매체를 변경하거나 기술적 우회 방안을 지속적으로 탐색 중.

## 3. Q&A 및 부연 설명
* **Q: 키즈나(Kizuna) MR 진행 시 특별히 주의할 점은?**
  * **A:** 요스타(Yostar) 측의 빌드 제공이 늦어질 수 있으나, 현재로서는 1월 말 진행을 목표로 인풋 준비에 집중해야 함. 특히 팀 내 이탈자가 많다는 보고가 있으므로 운영 리스크(HR 이슈) 체크가 필요함.
* **Q: 초보 실무자를 위한 가이드가 있는가?**
  * **A:** 현재 지혜 님이 실무자용 매뉴얼을 위키에 작성 중이며, 이를 통해 R&R과 프로세스 논의를 보강할 예정임.

## 4. 요약 및 결론
* 프로젝트 결과 기록의 통합 관리와 데이터 보안 정책 준수가 강조되었으며, AI 자동화 툴은 웹 생태계의 변화(봇 차단)에 따른 기술적 보완이 시급한 상태임.

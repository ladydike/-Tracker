# [세션 제목: MR AI Study - RAG 구현 및 MCP 기반 업무 자동화 심화]

## 1. 개요
* **분석 대상:** RAG(Retrieval-Augmented Generation) 시스템 구축, 문서 변환 도구(Docling), MCP 기반 서비스 구조화
* **세션 목적:** 개별 AI 프로젝트 진행 현황 공유 및 기술적 이슈(권한, 속도, 모델 연동) 해결 방안 논의

## 2. 상세 학습 내용 (주제별 정리)

### Section 1. [문서 전처리 및 데이터 가공 기술]
* **Docling 활용 문서 변환:**
  - PPT, PDF 등을 MD(Markdown) 형식으로 변환할 때 발생하는 양식 오류를 최소화함.
  - 이미지 내 텍스트는 캡처 및 설명이 가능하나, 이미지 자체를 완전하게 읽는 데는 한계가 있어 수동 보완(이미지 설명 요청 등)과 병행이 필요함.
* **벡터 DB(Vector DB) 및 임베딩:**
  - 방대한 데이터를 AI가 빠르게 검색할 수 있도록 벡터 DB로 변환하는 과정이 필수적임.
  - 단순 텍스트 검색을 넘어 AI가 이해하기 쉬운 형태로 데이터를 재구조화하여 검색 성능(RAG)을 향상시킴.

### Section 2. [RAG 시스템 및 컨플루언스 연동 이슈]
* **RAG(Retrieval-Augmented Generation) 구현:**
  - 회의록과 프로세스 위키를 학습시켜 실무 질문에 자동 답변하는 봇 구축 진행 중.
  - 현재는 로컬 MD 파일을 학습시키는 방식이며, 향후 실시간 연동을 목표로 함.
* **컨플루언스(Confluence) 권한 문제:**
  - AI 에이전트가 직접 컨플루언스 데이터를 읽어오는 과정에서 어드민/API 토큰 권한 이슈가 발생함.
  - 해결 방안: 정기적인 스케줄링(Batch)을 통해 로컬 DB로 데이터를 먼저 다운로드받은 후 처리하는 방식으로 우회 구현.

### Section 3. [MCP(Model Context Protocol) 기반 서비스 아키텍처]
* **기능별 프로젝트 분리(Modularization):**
  - 큰 기능을 하나의 프로젝트로 만들면 에러 추적이 어려우므로, 기능을 쪼개어 개별 MCP 서버로 구현할 것을 권장함.
  - 예: [데이터 크롤링] → [정제 및 외교적 어투 변환] → [웹 UI 출력]을 각각의 프로젝트로 분리하여 연결.
* **MCP의 3대 구성 요소:**
  - **도구(Tools):** AI가 실행할 수 있는 기능(글쓰기, 읽기 등) 정의.
  - **프롬프트(Prompts):** 일관된 결과물을 얻기 위한 사전 설정된 명령 세트.
  - **리소스(Resources):** AI가 참조할 데이터베이스나 파일 정보.

### Section 4. [실무 적용 사례: 자동화 워크플로우]
* **Daily 뉴스레터 자동화:**
  - 북미/중국 웹진 크롤링 → AI 기반 게임 관련성 및 중요도 필터링(0.4점 이상) → 국문 번역 → 아웃룩 자동 발송 파이프라인 구축.
* **회의록 팔로업 앱:**
  - 캘린더 형태의 UI에서 회의록을 요약하고 과제(To-do)를 자동으로 추출하여 추적하는 로컬 앱 개발.
* **슬랙 크리스(Chris) 연동:**
  - 매주 정해진 시간에 위키 템플릿을 생성하고, 명언 삽입 등 동기부여 요소를 포함한 자동 초안 작성 워크플로우 활용.

## 3. Q&A 및 부연 설명
* **Q: 클로드(Claude) API 비용을 절감하는 방법이 있나요?**
  - **A:** 모든 작업을 고성능 모델(Opus)로 처리하기보다, 단순 작업(키워드 추출)은 하이쿠(Haiku), 복잡한 분석은 소넷(Sonnet)으로 기능을 분리하여 호출하면 비용을 크게 줄일 수 있음.
* **Q: 비개발자도 MCP나 API 연동을 할 수 있을까요?**
  - **A:** 산업 디자이너 등 비개발자도 Cursor 등을 활용해 MCP 서버를 구축하는 사례가 많음. 기능을 쪼개어 접근하면 트러블 슈팅이 쉬워져 충분히 가능함.

## 4. 요약 및 결론
* **핵심 관통 한 줄 요약:** AI 프로젝트의 성공은 **'기능의 세분화(MCP)'와 '데이터의 최적화(RAG)'**에 달려 있으며, 복잡한 인프라 고민보다는 '일은 일로(Work as Work)' 접근하여 작은 모듈부터 성공시키는 것이 중요함.

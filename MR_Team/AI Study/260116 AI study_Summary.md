## 2. 상세 분석/학습 내용 (주제별 정리)

### Section 1. 업무 트래커 개발 사례 공유 (지혜)

#### 1.1 개발 배경 및 영감
- 이전 회의에서 민현 님이 공유한 **Cursor 폴더 카테고라이징 및 Cursor Rules 활용법**에 영감을 받아 개발
- **핵심 니즈**: 회의가 너무 많아서 팔로업이 어려운 상황 → 체계적인 관리 필요

#### 1.2 시스템 구조
**폴더 구조 설계**
- 회의록을 성격별로 카테고라이징하여 폴더 생성
- 클로바노트로 생성된 TXT 파일을 원본으로 활용
- 원본 TXT를 직접 요약하기 복잡하여 **Cursor Rules로 모드 시스템 구축**

**4가지 요약 모드**
- 회의록의 다양한 성격에 맞춰 4가지 모드로 분류
- 각 폴더별로 적용할 모드를 매핑
- 자동으로 요약 MD 파일이 생성되는 구조

#### 1.3 대시보드 기능
**회의록 뷰어**
- MR 미팅, AI 스터디, 대시보드 회의, MR 프로세스 회의, 딜리버리 리뷰, 게임 장르 분석, 게임톡 등 정리
- S2 미팅: 프로젝트별 카테고리로 확인 가능
- 원본 및 AI 요약본 동시 확인 가능
- **카드 형태 UI**: 초기 텍스트가 지저분하여 HTML 지식을 활용해 카드 형태로 개선

**회의 캘린더**
- 날짜별 회의 확인 가능 (예: 16일, 15일 회의 현황)
- 대시보드 화면에서 바로 요약 확인 가능
- 하루에 2건 이상 회의 시 선택 가능

**업무 일지 기능**
- 당일 업무 기재 가능
- 진행 상황 업데이트: 완료/진행 중 상태 관리
- **활용 예시**: 셀프 퍼블리싱 관련 대표님/자회사 관리팀 논의 내용 메모
- 퇴근 전 주요 업무 기재 → 나중에 검색/추적 가능
- 캘린더에 초록색 표시 = 업무 일지 입력됨

**AI 어시스턴트**
- **RAG 활용**: 원본 파일 + 요약 파일을 리소스로 검색
- 예시: "CEO 오피스 양윤상 님이 요청한 업무 내용 요약해줘" → 자동 검색 및 답변
- **Claude API** 사용
- 출처 명기 기능 포함 (로직 수정 필요한 부분 있음)

#### 1.4 기술적 구현
- 요약 자체를 앱에서 하려 했으나, 매번 API 호출 시 **비용 및 속도 문제**로 Cursor 내부에서 진행
- 파일명 기반 요약 요청 → Cursor Rules에 따라 자동 MD 파일 생성
- 개발 소요 시간: **하루 반** (원본 데이터 정리가 더 오래 걸림)

### Section 2. 팀 공유 및 데이터 관리 전략 (민현)

#### 2.1 공유 방식 제안
**두 가지 옵션**
1. **웹앱 배포**: 크레덴셜 별도 지정, 각자 사용
2. **소스코드 공유**: GitHub에 올려서 각자 로컬에서 커스터마이징 (권장)

#### 2.2 핵심 원칙: 원본과 후처리 분리

**권장 구조**
```
[원본 데이터] ← 공용 (구글 드라이브 등)
     ↓
[후처리/프로세스] ← 개인별 (각자 커스터마이징)
```

**분리해야 하는 이유**
- **의존성 문제**: 통합 시 개인별 버전 꼬임 발생
- **변동성 관리**: 후처리 규칙은 변경/고도화될 수 있음
- **오염 방지**: 후처리 결과가 원본에 연결되면 데이터 오염 발생
- **장기적 안정성**: 1~2년 후에도 문제없는 구조

**구체적 방안**
- 원본 소스: 구글 드라이브에 공용 폴더 생성 → 회의록 한 곳에 적재
- 로컬 싱크로 자동 동기화
- 툴/프로세스: 개인별로 프라이빗하게 운영

#### 2.3 공용 대시보드에 대한 의견
- **필요성 의문**: "공용으로 볼 필요가 있을까?"
- **의존성의 폐해**: 좋아서 하는 것들이 오히려 장벽이 되거나, 개인별 맞춤화를 막음
- **권장**: 명확한 효용 가치 발견 전까지 통합화 지양

#### 2.4 AI 시대의 데이터 가치

**핵심 메시지**
> "AI 시대에 제일 중요한 건 **원본 데이터를 깔끔하게 보존**하는 것"

- 앱 만들기, 관리하기는 "딸깍"으로 거의 다 되는 추세
- **원본 데이터의 가치가 엄청나게 상승**
- 처리/가공의 가치는 거의 제로에 수렴
- AI 툴은 계속 좋아질 것 → 현재 툴에 연연할 필요 없음

### Section 3. Git/버전 관리 관련 조언 (민현)

#### 3.1 비엔지니어의 코드 협업 위험성

**일반적인 Git 워크플로우**
1. 메인 브랜치에서 개인 브랜치 생성
2. 개인 브랜치에서 작업
3. 좋은 기능은 메인 브랜치에 머지

**비엔지니어가 하면 안 되는 이유**
- 메인 브랜치 직접 수정 → **빌드 깨짐** → 다른 사람 것도 깨짐
- 프로그래밍 조직 불문율: "자기가 커밋해서 깨지면 니가 고쳐"
- 커밋 탈 때 누가 뭘 바꿨는지 다 추적 가능

**해결책**
- 자신 없으면 피어 리뷰 (시니어와 코드 리뷰)
- 또는 애초에 메인 브랜치 접근 제한

#### 3.2 권장 방식
- 기능 추가 시 프롬프트 공개하면 각자 해결 가능
- 개별적으로 기능 만들면 공유: "이런 기능 추가했는데 필요하시면 소스 코드 드릴게요"

### Section 6. AI 시대 업무 방식 변화

#### 6.1 프롬프트의 역사적 가치
**BG님의 새로운 요구사항**
- AI 결과물 보고 시 **원본 대화(프롬프트) 요청**
- "결론만 보지 않고 어떤 과정을 거쳐서 나왔는지 보고 싶다"
- 경영진에게도 동일하게 적용

**마리아 님 사례**
- ChatGPT 대화 링크를 그냥 던지는 방식으로 커뮤니케이션

#### 6.2 역방향 프롬프트 작성법 (핵심 인사이트)

**기존 방식 (비효율적)**
```
원본 → "이렇게 바꿔줘" 프롬프트 → 원하는 결과 안 나옴
```

**권장 방식**
```
원본 + 원하는 결과물(베스트 프랙티스) → "둘 사이 갭이 뭔지 알려줘" → 프롬프트 도출
```

**비유**: 사람 가르치는 것과 동일
- 베스트 프랙티스 보여주고 "비슷하게 만들어봐"가 더 효과적
- "이렇게 해보세요, 저렇게 해보세요"는 무한 삽질

#### 6.3 회의록 교정 DB 구축 제안

**목표**
- 공동의 교정 디렉션 파일 만들기
- 교정용 규칙을 주기적으로 업데이트

**방법**
1. 클로드 프로젝트 생성
2. 교정 전/후 파일을 비포/애프터 프리픽스로 업로드
3. 한 달간 데이터 축적
4. AI가 패턴 분석하여 교정 규칙 도출

#### 6.4 경영진의 기대 변화

**링크드인 트렌드**
> "수행이 너무 빠르기 때문에 계획을 잘 짜던 시대는 끝났다. 하루 이틀 만에 해봐서 결과를 보는 게 중요하다"

**해석**
- 경영진 입장: "하루면 할 수 있잖아"로 질문이 바뀜
- 가치 판단 전에 일단 해보겠다는 의미
- **쓰자띠 검증(빠른 검증)** 시대

#### 6.5 보고 방식의 변화

**과거**: "왜 보고 문서에 이 내용이 없어요?"
**현재**: "왜 데이터 스토리지에 이 데이터가 없어요?"

- 임원이 직접 AI에 질문해서 답을 얻는 방식으로 전환
- 심플리파이보다 **학습 자료/어벤져스에 데이터가 깔려 있는 것**이 중요
- 정보를 받아들이고 처리하는 방식 자체가 변화 중

## 5. 요약 및 결론

1. **업무 트래커 완성**: 지혜 님이 Cursor Rules를 활용한 회의록 관리 시스템을 완성했으며, 대시보드/업무일지/AI 어시스턴트 기능을 포함한 종합 업무 관리 도구로 발전시켰다. 개발 소요 시간은 하루 반 정도로, AI 도구를 활용한 빠른 프로토타이핑의 좋은 사례다.

2. **데이터 관리 원칙**: 민현 님은 원본 데이터와 후처리를 분리하는 것이 핵심이라고 강조했다. 공용으로 관리해야 할 것은 원본 데이터(TXT 파일)뿐이며, 요약/가공은 개인별로 운영하는 것이 장기적으로 안전하다. AI 시대에는 처리/가공의 가치는 거의 제로에 수렴하고, 원본 데이터의 가치가 엄청나게 상승했다.

3. **클로드 프로젝트 활용**: 긴급 투자 검토 건에서 클로드 프로젝트를 활용해 48시간 만에 보고를 완료한 사례가 공유되었다. 핵심은 프로젝트에 자료를 축적하고, 최종 보고 시 프로젝트 권한을 부여하여 경영진이 직접 질의할 수 있게 하는 방식이다.

4. **프롬프트 작성 팁**: "이렇게 바꿔줘"보다 원본과 원하는 결과물(베스트 프랙티스)을 함께 제시하고 "둘 사이 갭을 분석해줘"라고 요청하는 역방향 접근이 더 효과적이다. 회의록 교정 규칙도 이 방식으로 도출할 수 있다.

5. **AI 시대 업무 환경**: 경영진의 기대가 "왜 못 해?"에서 "왜 하루 만에 못 해?"로 바뀌고 있으며, 보고 방식도 문서 중심에서 데이터/AI 질의 중심으로 전환 중이다. 현재 AI 도구에 연연하지 말고, 갈아타기 쉬운 구조(깔끔한 원본 데이터)를 만드는 것이 중요하다.
